# ShelfChef — Product Specification v1

## 1. Purpose

ShelfChef is a household pantry and meal-planning app. The core value loop:

> Track what you have → discover what you can cook → plan meals → generate a shopping list → mark as cooked → pantry updates.

v1 ships the full loop end-to-end.

---

## 2. Users & Households

### 2.1 Household model
- A **Household** is the shared unit. Every pantry item, recipe, meal plan, and shopping list belongs to a household.
- A **User** belongs to exactly one household at a time (they can leave and join another).
- **Roles**: Flat — all members have equal read/write access. No role hierarchy in v1.

### 2.2 Authentication
- **Email + password** — bcrypt hashed, email verification required before first login.
- **OAuth** — Google and GitHub via Passport.js. On first OAuth login, user is created automatically.
- **Sessions** — Stateless JWTs (access token + refresh token). Access token: 15 min TTL. Refresh token: 7 days, stored in httpOnly cookie.
- JWT secret in environment variable `JWT_SECRET`.

### 2.3 Household lifecycle
- On registration, user can **create** a new household or **join** an existing one via invite code.
- Invite codes: 8-character alphanumeric, generated by any household member, expire after 48 hours. Stored as `HouseholdInvite` documents.
- Any member can generate a new invite link. Any member can remove another member or disband the household (flat — no owner distinction).
- Disbanding a household: soft-delete all household data (mark `deletedAt`). Hard-delete job deferred to v2.

## 3. Core Entities & Relationships

```
User ──belongs to──► Household
                         │
              ┌──────────┼────────────┬──────────────┐
              ▼          ▼            ▼              ▼
         PantryItem   Recipe       MealPlan      ShoppingList
                         │
                    RecipeIngredient (embedded)
```

**Key cardinalities:**
- Household → PantryItems: 1:N
- Household → Recipes (custom): 1:N
- Household → MealPlan: 1:1 (one active plan per week; historical plans retained)
- MealPlan → MealSlots: 1:7 days × 3 slots (Breakfast/Lunch/Dinner)
- MealSlot → Recipe: 1:0..1 (one recipe per slot, or empty)
- Household → ShoppingList: 1:1 active list

## 4. MongoDB Data Model

### 4.1 User
```ts
{
  _id: ObjectId,
  email: string,                        // unique, indexed
  passwordHash: string | null,          // null for OAuth-only users
  oauthProviders: [{
    provider: 'google' | 'github',
    providerId: string,
  }],
  emailVerified: boolean,
  household: ObjectId | null,           // ref: Household
  refreshTokenHash: string | null,      // bcrypt hash of current refresh token
  createdAt: Date,
  updatedAt: Date,
}
```

### 4.2 Household
```ts
{
  _id: ObjectId,
  name: string,
  members: ObjectId[],                  // ref: User
  createdAt: Date,
  updatedAt: Date,
  deletedAt: Date | null,
}
```

### 4.3 HouseholdInvite
```ts
{
  _id: ObjectId,
  household: ObjectId,
  code: string,                         // 8-char alphanumeric, unique indexed
  createdBy: ObjectId,                  // ref: User
  expiresAt: Date,                      // createdAt + 48h
  usedBy: ObjectId | null,             // ref: User (single-use)
  createdAt: Date,
}
```

### 4.4 PantryItem
```ts
{
  _id: ObjectId,
  household: ObjectId,                  // indexed
  name: string,                         // user-entered, normalized lowercase for matching
  quantity: number,                     // >= 0
  unit: CanonicalUnit,                  // enum (see §4.8)
  spoonacularIngredientId: number | null,  // for alias matching
  aliases: string[],                    // known name variants (alias table)
  addedBy: ObjectId,
  createdAt: Date,
  updatedAt: Date,
}
```

### 4.5 Recipe
```ts
{
  _id: ObjectId,
  source: 'spoonacular' | 'custom',
  spoonacularId: number | null,         // null for custom; indexed (sparse)
  household: ObjectId | null,           // null = spoonacular (cached globally); set for custom
  title: string,
  description: string,
  imageUrl: string | null,
  servings: number,
  readyInMinutes: number | null,
  ingredients: [{
    name: string,
    quantity: number,
    unit: CanonicalUnit,
    spoonacularIngredientId: number | null,
  }],
  instructions: string[],               // ordered steps
  cachedAt: Date | null,               // when Spoonacular data was last fetched
  createdAt: Date,
  updatedAt: Date,
}
```

### 4.6 MealPlan
```ts
{
  _id: ObjectId,
  household: ObjectId,
  weekStart: Date,                      // Monday 00:00 UTC; unique per household
  slots: [{
    date: Date,                         // ISO date of the day
    mealType: 'breakfast' | 'lunch' | 'dinner',
    recipe: ObjectId | null,            // ref: Recipe
  }],
  createdAt: Date,
  updatedAt: Date,
}
// Compound unique index: { household, weekStart }
// One-recipe-per-slot enforced in service layer
```

### 4.7 ShoppingList
```ts
{
  _id: ObjectId,
  household: ObjectId,                  // unique index (one active list)
  items: [{
    _id: ObjectId,
    name: string,
    quantity: number,
    unit: CanonicalUnit,
    recipeSource: ObjectId | null,      // which recipe this came from
    checked: boolean,
    addedAt: Date,
  }],
  updatedAt: Date,
}
```

### 4.8 Canonical Unit Enum
```
// Volume
'ml' | 'l' | 'tsp' | 'tbsp' | 'cup' | 'fl_oz'
// Weight
'g' | 'kg' | 'oz' | 'lb'
// Count
'piece' | 'pinch' | 'slice'
// Freeform fallback
'unit'
```
Conversion factors stored in `shared/constants/unitConversions.ts`. All internal math in metric (ml/g). Display always metric in v1 (user-preferred system is v2).

## 6. Business Logic

### 6.1 Recipe Match Score
```
matchScore = ingredientsAvailable / totalIngredients  (0.0–1.0)
```
- "Available" = pantry has an item whose name or alias matches the recipe ingredient, and quantity in the same unit family is >= required after conversion.
- Response shape: `{ score, available: Ingredient[], missing: Ingredient[] }`.
- Frontend shows e.g. "8/10 ingredients on hand".

### 6.2 Ingredient Matching (Alias Table)
- Priority order:
  1. `spoonacularIngredientId` exact match (when both pantry item and recipe ingredient carry an ID).
  2. `PantryItem.aliases[]` exact match (normalized lowercase).
  3. Substring fallback: `'flour'` matches `'all-purpose flour'`.
- Alias management: "also known as" field on the pantry item edit form.
- Cross-unit-family comparisons (e.g. `g` vs `piece`) are blocked and surfaced as "unit mismatch" in the match report.

### 6.3 Suggested Deduction Flow
1. `POST /meal-plan/cook` → returns `DeductionProposal`:
   ```ts
   { items: [{ pantryItemId, name, currentQty, deductQty, unit, newQty }] }
   ```
2. Client shows confirmation modal. User can adjust quantities or uncheck items.
3. `POST /meal-plan/deduct` applies atomic `$inc` ops. Quantities clamped to 0 minimum.
4. Unmatched recipe ingredients are silently skipped.

### 6.4 Shopping List Generation
- `POST /shopping-list/from-meal-plan` aggregates all ingredients from the current week's non-empty slots.
- Does **not** subtract pantry stock — full ingredient list always added.
- Duplicate detection: same name + unit → quantities summed. New items appended to existing list.

### 6.5 Spoonacular Caching
- Search results upserted into MongoDB as `Recipe` docs (`source: 'spoonacular'`, `cachedAt: now`).
- Cache TTL: 24 hours. Requests within TTL return DB results without hitting Spoonacular.
- On Spoonacular error (non-2xx, timeout, 429): return stale cached results with `{ stale: true }` in response. Client shows "Results may be outdated" banner.
- If no cached data exists at all: return 503.

## 7. Validation Architecture

| Layer | Tool | Responsibility |
|-------|------|----------------|
| API boundary | Zod (`shared/schemas/`) | Shape, type, range. Returns 400 with `errors[]`. |
| Service layer | TypeScript guard clauses | Business rules (slot occupied, not member's recipe, qty negative). Returns 422. |
| DB layer | Mongoose validators | Enum values, required fields, numeric min. Last safety net. |

Frontend imports the same Zod schemas for instant UX validation — no duplication.

## 9. Error Handling

| HTTP | When |
|------|------|
| 400 | Zod validation failure |
| 401 | Missing or invalid JWT |
| 403 | Not a household member, or modifying another household's data |
| 404 | Resource not found |
| 409 | Conflict (invite code used, email taken) |
| 422 | Business rule violation (slot occupied, etc.) |
| 503 | Spoonacular down and no cached data |

All error responses: `{ error: string, code?: string, details?: object }`.

## 10. Testing Strategy

### Unit (Vitest)
- Service functions in isolation (mock Mongoose).
- Unit-conversion utilities.
- Match score algorithm.
- Zod schema edge cases.

### Integration (Vitest + supertest)
- Full HTTP → service → MongoDB (`mongodb-memory-server`).
- One file per route group.
- Auth middleware, household membership, CRUD happy paths + error cases.

### E2E (Playwright)
1. Register → create household → add pantry item
2. Search Spoonacular recipe → view match score
3. Add recipe to meal plan → generate shopping list → check off → move to pantry
4. Invite second user → joins → sees same pantry

### Coverage
- Vitest threshold: **80% lines** on `server/services/` and `shared/`.
- E2E: qualitative flow coverage only.

## 14. Risks & Mitigations

| # | Risk | Mitigation |
|---|------|-----------|
| 1 | Spoonacular ingredient names rarely match pantry entries | Alias table + substring fallback; seed common aliases |
| 2 | Cross-unit-family conversion impossible (g vs piece) | Block comparison; surface as "unit mismatch" in match report |
| 3 | Concurrent pantry edits (last-write-wins) | Accepted for v1; documented limitation |
| 4 | Spoonacular free tier: 150 req/day | 24h search cache; individual recipe cached indefinitely |
| 5 | Email delivery for verification | Resend or Nodemailer+SMTP; provider swappable via env var |
